SLACK's API

This week I have worked with SLACK's API.

API is the acronym for Application Programming Interface, which is a software intermediary that allows two applications to talk to each other.

SLACK's API works with web requests. It has several types of layers and functions.

I'll mention some of them just below.

  • RTM: Real-time messaging. It works with WebSockets. The documentation recommends avoiding using them. It is a legacy that works fine for small teams, but as users increment, its performance plummeted.

  • Events: They pick events of interest. Slack will send it through HTTP. They are focused on actions (for example, clicked buttons).

  • Web-API: It is used to receive information from outside (another app or site).

  • Conversation: Subsets of Web-API that calls slack directly.

Slack Features

The slack commands work almost everywhere, channels, conversations, group conversation.

Block-kit. It provides all the GUI elements. They are defined by JSON

There exist two ways of posting messages.

  1. Incoming webhook

  2. Using chat.postMessage function

Incoming webhook

They are the simplest but least flexible they can generate post messages on channels. They are used via single-use URLs. They also can send notifications from other servers and read data from slack.

chat.postMessage

They call SLACK's API through HTTP. They can post everywhere and receive responses.

Ephemeralmessages

Messages in channels that only show to specific users, disappear with time. There is no guarantee that the messages reach their intended if they are not logged at the time.

Dialogs

They can only be created in response to a direct user action and supports:

  • Single line text entries

  • Dropdown select

  • Longer text area (limited to 3000 characters)
